## 一句话结论

Claude Code 的 Agentic Loop 不是“模型反复思考”的抽象说法，而是 `src/query.ts` 里一台显式维护 `State`、在每一轮末尾决定 `continue` 还是 `return` 的多轮控制器；真正让它可研究的，不是 `while (true)` 本身，而是每一种继续原因都会改写下一轮真正看到的消息面。

## 30 秒摘要

这页讲的是多轮主循环这件事本身，不是某一轮里具体每个局部变量怎么变。`queryLoop()` 定义在 `src/query.ts:241`，真正进入循环在 `src/query.ts:307`。每轮先做消息预算与压缩，再调用模型流式采样，再根据 `tool_use`、恢复型错误、stop hook、token budget 和 `maxTurns` 决定是结束，还是把新状态拼回去跑下一轮。

如果你只记一个判断标准，就是这句：**Claude Code 不是“模型先计划完，再批量执行”；它是每拿到一批新观察，就重写一次下一轮输入。**

## 状态标签总览

| 子路径 / 机制 | 状态标签 | 当前含义 |
|---|---|---|
| `queryLoop()` 主循环、工具继续、终止条件 | `external build active` | 当前 external build 的真实多轮控制器 |
| `StreamingToolExecutor` 边流式边起工具 | `external build active` | 当前仓库真实活跃 |
| `max_output_tokens` 扩槽与续写恢复 | `external build active` | 当前 `query.ts` 明确实现 |
| `FallbackTriggeredError` 模型 fallback | `external build active` | 路径真实存在，是否触发取决于 fallback model 配置 |
| stop hook 阻塞后继续 | `external build active` | 当前 external build 真实路径 |
| token budget continuation / diminishing returns | `feature-gated` | 依赖 `feature('TOKEN_BUDGET')` |
| context collapse overflow drain | `feature-gated` | 依赖 `feature('CONTEXT_COLLAPSE')` |
| reactive compact overflow/media 恢复 | `stubbed/removed` | `query.ts` 保留分支，但当前仓库实现文件是 stub，不应写成 live 能力 |

## 为什么存在

如果系统只是“一问一答”，完全没必要做这么重的循环控制。之所以会长成现在这样，是因为 Claude Code 必须同时解决四件事：

- 工具结果不可预知。命令输出、文件内容、权限拦截、MCP 返回值都只能在上一轮结束后才知道。
- 上下文不是静态的。每一轮都可能因为工具结果过大、历史太长或缓存编辑而改变下一轮输入面。
- 错误不总该直接报废整轮。`max_output_tokens`、stream fallback、模型 fallback、stop hook 阻塞都更适合被改写成“继续一次”。
- 主循环还承担控制平面职责。工具执行完以后，不只是把 `tool_result` 接回去，它还会插入附件、消费队列消息、吸收 memory prefetch、刷新工具池，再生成下一轮状态。

所以 Agentic Loop 的本质不是“让 AI 多想几次”，而是让系统把真实世界的新观测不断并回 prompt。

## 正常链路

```mermaid
flowchart TD
  start["state.messages"] --> boundary["getMessagesAfterCompactBoundary()"]
  boundary --> budget["applyToolResultBudget()"]
  budget --> snip["snipCompactIfNeeded() if enabled"]
  snip --> micro["microcompact()"]
  micro --> collapse["contextCollapse.applyCollapsesIfNeeded() if enabled"]
  collapse --> auto["autocompact()"]
  auto --> call["deps.callModel() stream"]
  call --> collect["收 assistant / tool_use / partial tool results"]
  collect --> follow{"needsFollowUp?"}
  follow -->|no| recover["413 / media / max_output_tokens 恢复判定"]
  recover --> hooks["stop hooks / token budget / completed"]
  follow -->|yes| tools["StreamingToolExecutor or runTools()"]
  tools --> sidecar["附件、memory、队列、refreshTools"]
  sidecar --> next["构造 next State"]
  next --> start
```

这条图里最容易被漏掉的，不是 `tool_use`，而是两件事：

- 预处理链发生在每一轮，而不是整个会话一开始只跑一次。
- 工具执行之后还有一段“控制平面收尾”，下一轮看到的不只是 `tool_result`，还可能包括附件、队列消息、memory 和刷新后的工具列表。

## 关键结构 / 状态

`query.ts` 顶部定义的 `State` 已经把这台循环真正关心什么写得很直白：

| 字段 | 作用 | 为什么重要 |
|---|---|---|
| `messages` | 下一轮真正发给模型的消息历史 | 主循环不是保存“日志”，而是在维护可继续的 prompt 面 |
| `toolUseContext` | 工具池、权限、AppState、abort controller 等总入口 | 工具执行和模型调用共用这份上下文 |
| `autoCompactTracking` | 跟踪最近一次 compact 与失败计数 | 决定 autocompact 何时再触发 |
| `maxOutputTokensRecoveryCount` | 限制续写恢复次数 | 防止 `max_output_tokens` 无限重试 |
| `hasAttemptedReactiveCompact` | 记录是否已经试过 reactive compact | 防止 prompt-too-long / media 错误死循环 |
| `maxOutputTokensOverride` | 暂时把输出槽从默认值抬高 | 这是“扩槽重试”和“注入续写消息”之间的分水岭 |
| `pendingToolUseSummary` | 把工具摘要异步生成并延后到下一轮前输出 | 说明循环里还有并行的旁路工作 |
| `stopHookActive` | 标记当前是不是 stop hook 驱动的重跑 | 影响后续 stop hook 处理 |
| `turnCount` | 已经跑了多少轮 | 用于 `maxTurns` 和可观测性 |
| `transition` | 上一轮为什么继续 | 这是理解恢复路径最关键的证据字段 |

除了 `State`，还要盯住两类“继续原因”：

| `transition.reason` | 属于哪类继续 | 当前意义 |
|---|---|---|
| `next_turn` | 正常继续 | 有 `tool_use`，工具结果回来后进入下一轮 |
| `max_output_tokens_escalate` | 恢复型继续 | 同一请求先抬高输出槽再试 |
| `max_output_tokens_recovery` | 恢复型继续 | 追加 meta message 要求模型直接续写 |
| `collapse_drain_retry` | 恢复型继续 | 先排空 staged collapses 再试，但依赖 feature gate |
| `reactive_compact_retry` | 恢复型继续 | 设计存在，但当前仓库实现不应写成 live |
| `stop_hook_blocking` | 策略纠偏继续 | 先拼回 blocking errors 再让模型重答 |
| `token_budget_continuation` | 成本控制继续 | 预算提醒后再给一轮机会 |

这也是为什么这页必须和 [单轮状态机](/docs/conversation/single-turn-state-machine) 分开读：单轮页讲一次迭代内部怎么走，这页讲为什么系统会一轮又一轮地改写自身状态。

## 正常链路拆开看

从源码顺序看，一轮主循环的“正常路径”大致可以拆成五段：

1. `src/query.ts:365-468` 先从 compact boundary 之后取可见历史，然后依次跑 `applyToolResultBudget()`、`microcompact()`、`contextCollapse.applyCollapsesIfNeeded()`、`autocompact()`。这一步决定本轮到底拿什么去问模型。
2. `src/query.ts:551-866` 初始化 `assistantMessages`、`toolUseBlocks`、`needsFollowUp` 和 `StreamingToolExecutor`，再进入 `deps.callModel()` 的流式循环。
3. 流中一旦发现 `tool_use`，`needsFollowUp` 立刻变成 `true`，并尽可能让 `StreamingToolExecutor` 先把工具跑起来；所以工具并发不是“流结束之后才开始”，而是已经嵌进流里。
4. 如果本轮最后 `needsFollowUp === false`，就进入 `src/query.ts:1065-1360` 的恢复、stop hook、token budget 与完成路径。
5. 如果 `needsFollowUp === true`，就进入 `src/query.ts:1383-1730`：取回剩余工具结果，追加附件与 memory，刷新工具池，最后把 `messages + assistantMessages + toolResults` 拼成下一轮 `state.messages`。

这五段一起看，才会明白 Agentic Loop 为什么更像“控制平面”而不是“简单递归调用”。

## 一个实际例子

假设用户说：“检查文档站导出为什么卡住，并修掉会导致页面 500 的问题。”

一条典型的多轮路径会长这样：

1. 第 1 轮，模型先调用 `Glob` 或 `Grep` 找到可疑页面和导出脚本。因为出现了 `tool_use`，本轮不会结束。
2. 第 2 轮，模型基于搜索结果调用 `Read`、`Bash`，读 `the-loop.mdx` 和导出日志。工具结果被接回消息面，形成新的上下文。
3. 第 3 轮，模型发现页面结构损坏，调用 `Edit` 修正文档，再调用 `Bash` 重跑导出。此时循环里除了 `tool_result`，还可能额外吸收“文件已编辑”的附件消息。
4. 第 4 轮，如果导出通过、模型不再发 `tool_use`，就进入 stop hook 与完成路径，最后返回 `completed`。

这里真正关键的不是“跑了 4 轮”，而是每一轮看到的世界都不同：

- 第 1 轮看到的是原始用户需求。
- 第 2 轮看到的是代码搜索结果。
- 第 3 轮看到的是具体文件内容和报错。
- 第 4 轮看到的是修复后的验证结果。

这正是 Agentic Loop 必须存在的理由。

## 失败与恢复

多轮循环不是只有“有工具就继续”这一条路，`query.ts` 还把多种失败改写成了继续条件。

| 问题 | 当前仓库的处理方式 | 会不会继续 |
|---|---|---|
| `max_output_tokens` | 先尝试把输出槽提升到 `ESCALATED_MAX_TOKENS`，不够再注入“直接续写”meta message | 会 |
| stream 中途 fallback | tombstone 已收集的半成品 assistant messages，重建 `StreamingToolExecutor` 后重试 | 会 |
| repeated 529 触发模型 fallback | 捕获 `FallbackTriggeredError`，切换到 fallback model，再清理旧 tool use 状态 | 会 |
| stop hook 给出 blocking errors | 把 blocking errors 追加到消息面后重跑 | 会 |
| token budget 判断还能继续但需要收尾提醒 | 注入 nudge message 再给一轮 | 会，且是 feature-gated |
| 413 prompt-too-long | 先看 context collapse drain，再看 reactive compact；但当前仓库不应把 reactive compact 写成 live 路径 | 设计上会，当前仓库要保守写 |
| 图片/媒体尺寸错误 | 会先尝试 reactive compact 媒体恢复分支；当前仓库同样不能写成稳定 live 能力 | 设计上会，当前仓库要保守写 |

可以把恢复体系理解成一个原则：**只要错误还能转译成“下一轮能理解的状态”，循环就尽量不直接报废。**

一个最典型的恢复例子是 `max_output_tokens`：

1. 流式阶段先把这个错误暂扣，不立刻对外抛。
2. 如果本轮还没用过 `maxOutputTokensOverride`，先升级输出槽再跑同一请求。
3. 如果升级后还不够，就在消息尾部加一条 meta message，明确要求模型从中断处继续，不要道歉也不要 recap。
4. 只有恢复次数耗尽，才真正把 withheld error 放出来。

这条链路说明，Claude Code 把“回答太长”视为资源约束，不是推理失败。

## 边界与误读

- Agentic Loop 不是 `QueryEngine` 的同义词。多轮主循环真正落在 `src/query.ts`；有些入口会在外层包装，但 loop 本体在这里。
- `while (true)` 不等于“理论上无限跑”。`completed`、`aborted_streaming`、`aborted_tools`、`max_turns`、`hook_stopped`、`prompt_too_long` 等都会终止。
- `needsFollowUp === false` 也不一定立刻结束，因为后面还可能进入 `max_output_tokens`、stop hook、token budget 等继续路径。
- `tool_use` 不是唯一继续原因。恢复型继续和策略纠偏继续同样真实存在。
- 看到 `reactiveCompact.tryReactiveCompact()`，不能直接写成“当前 external build 已稳定支持即时压缩恢复”；当前 reverse-engineered 仓库里实现文件是 stub。
- `contextCollapse` 相关恢复也不能写成默认 live 路径，因为它挂在 `feature('CONTEXT_COLLAPSE')` 后面。
- “流式工具执行”也不是另一个独立回合系统；它只是把部分工具执行提前嵌进本轮 stream。

## 场景变体

| 场景 | 主循环最关键的分支 |
|---|---|
| 普通问答 | 无 `tool_use`，直接走 completed |
| 文件修改或调试任务 | `next_turn` 是主路径，工具结果决定下一轮 |
| 长回答 / 长计划 | `max_output_tokens_escalate` 与 `max_output_tokens_recovery` 更重要 |
| 高负载或 provider 问题 | streaming fallback、model fallback 更重要 |
| 有 stop hooks 的治理环境 | `stop_hook_blocking` 会把“当前回答不合规”转成下一轮输入 |
| 预算敏感环境 | `token_budget_continuation` 和 diminishing returns 会提前收尾 |
| 研究内部机制 | 可以分析 collapse / reactive compact 分支，但必须把 live 状态写清楚 |

## 一句话结论

GrowthBook 在当前仓库里不是第一层门，而是 **build-time feature 之后的第二层运行时决策器**：代码先得活着，它才有机会决定“当前会话要不要看到这条能力线”。

## 状态标签总览

| 主题 | 当前状态 | 说明 |
|---|---|---|
| GrowthBook 客户端与属性装配 | `external build active` | 不是空壳，当前仓库有真实初始化与刷新逻辑 |
| 阻塞初始化读取 | `external build active` | `initializeGrowthBook()` / `getFeatureValue_DEPRECATED()` 仍存在 |
| 非阻塞缓存读取 | `external build active` | `getFeatureValue_CACHED_MAY_BE_STALE()` 是 startup-critical 路径首选 |
| ant 专用 override 能力 | `ant-only` | `CLAUDE_INTERNAL_FC_OVERRIDES`、config overrides 只对 ant 世界开放 |
| feature gate 之外的 dormant branch 激活 | `feature-gated` 的前提之上 | GrowthBook 不能把 build 没带进来的代码“复活” |

## 为什么存在

如果所有能力判断都放在 `feature('...')` 里，会有两个问题：

1. build 一改就要重新发版，无法对同一份活跃代码做细粒度放量、回滚和分群。
2. 某些启动关键路径需要“立刻给出一个值”，不能每次都阻塞在远程实验系统初始化上。

所以当前仓库把“代码是否在当前 build 存在”和“当前用户/组织/环境是否应当看到它”拆成了两层：

- 第一层：`feature()` 决定这段代码是否属于当前 external build。
- 第二层：GrowthBook 决定这段已经活着的代码，要不要对这次会话开放。

## 正常链路

```mermaid
flowchart LR
  build["feature() / active build"] --> attrs["getUserAttributes()"]
  attrs --> init["initializeGrowthBook()"]
  init --> payload["remote eval payload / disk cache"]
  payload --> read["getFeatureValue_* / getDynamicConfig_*"]
  read --> refresh["periodic refresh / refreshed.emit()"]
  refresh --> surface["会话能力面更新"]
```

## 关键结构 / 状态

| 结构 | 作用 | 关键点 |
|---|---|---|
| `isGrowthBookEnabled()` | 决定是否启用 GB | 依赖 1P event logging，不是无条件 always-on |
| `initializeGrowthBook()` | 阻塞直到客户端 ready | 适合需要强一致值的路径，但会拖慢启动 |
| `getFeatureValue_CACHED_MAY_BE_STALE()` | 立即从内存或磁盘缓存返回 | 当前仓库明确把它作为 startup-critical 默认读取方式 |
| `remoteEvalFeatureValues` | 进程内权威值缓存 | refresh 后会重建，不只是磁盘缓存副本 |
| `cachedGrowthBookFeatures` | 跨进程磁盘缓存 | 允许下一次启动即时读取，但可能陈旧 |
| `onGrowthBookRefresh()` | 通知长期对象重建 | 用于那些把 gate 值烘焙进长寿命对象的系统 |

这里最容易被低估的一点是：GrowthBook 不是“只在网络请求完成后读一次值”。它同时维护：

- 阻塞初始化路径
- 非阻塞缓存路径
- 周期性轻刷新路径
- auth 改变后的重建路径

也就是说，它不仅是开关系统，还是 **一套“如何在长会话里保持 gate 新鲜度”的机制**。

## 一个端到端例子

假设某个能力已经通过 build-time gate 进入当前 external build，但是否默认对用户开放还要看运行时条件：

1. 启动早期代码走 `getFeatureValue_CACHED_MAY_BE_STALE()`，先从内存或磁盘里拿一个可能陈旧、但足够快的值。
2. 随后 `initializeGrowthBook()` 完成，remote eval payload 填充 `remoteEvalFeatureValues`。
3. 如果实验数据已经可用，先前累积在 `pendingExposures` 里的曝光记录会补记。
4. 长会话里再通过 `setupPeriodicGrowthBookRefresh()` 做轻刷新，把新值同步到磁盘和订阅者。

这条链的价值在于：**启动时不必全阻塞，运行中也不必永远相信旧缓存**。

## 为什么不是更简单的设计

一种更简单的方案是：每次都只在初始化时阻塞拉取远程实验值。

当前仓库没有这么做，原因很实际：

- 启动太早拉网络会拖慢 CLI 首屏。
- trust 尚未建立时，不应该为了实验系统去提前触发 auth。
- 有些长寿命系统需要在运行中感知 refresh，而不是直到下一次重启。

所以它才会同时保留 `_BLOCKS_ON_INIT` 和 `_CACHED_MAY_BE_STALE` 两类 API。这不是命名冗余，而是对不同时延/一致性需求的显式分工。

## 失败与恢复

| 失败场景 | 当前表现 | 恢复方式 |
|---|---|---|
| trust 或 auth 未建立 | GB 上下文不完整，只能给默认值或缓存值 | 等 trust/auth 就绪后重建客户端 |
| 磁盘缓存陈旧 | 启动早期读到过期实验值 | 初始化完成或 periodic refresh 后会纠正 |
| 客户端创建时没有 auth，后来 auth 才可用 | 早期属性不完整 | `initializeGrowthBook()` 会检测并触发 reset / recreate |
| 把 GB 当第一层 gate | 误以为 GB=true 就能让 dormant branch 活过来 | 先回到 `feature()` 判活，再谈 GB |
| 长寿命对象烘焙了旧 gate 值 | 会话里行为不随 refresh 更新 | 通过 `onGrowthBookRefresh()` 订阅重建 |

## 边界与误读

<Warning>
GrowthBook 无法“复活” build 没带进来的代码。它只能在已经活着的路径上继续做运行时决策。
</Warning>

- 不要把 GrowthBook 写成 feature flag 的替代物；它是后一层。
- 不要把 `_CACHED_MAY_BE_STALE` 读法写成“不可靠的临时 hack”；这是当前仓库明确偏爱的启动路径。
- 不要忽略 ant-only overrides；它们会影响你如何理解内部实验与公开世界的一致性。
- 不要把“有 GB 代码”误写成“所有 gate 都依赖远程网络实时返回”。

## 场景变体

| 场景 | GrowthBook 的角色 |
|---|---|
| 启动关键路径 | 立即给缓存值，避免阻塞 |
| 鉴权后长期会话 | 周期刷新、订阅更新、逐步变新 |
| ant 内部调试 | env/config override 可直接改 gate 结果 |
| external build 事实判断 | 仍必须先看 build-time gate |

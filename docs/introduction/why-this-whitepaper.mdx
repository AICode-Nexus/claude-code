## 一句话结论

这套文档站的目标不是教你“如何开始用 Claude Code”，而是告诉你“当前这个仓库到底会运行什么、为什么这样设计、哪些地方最容易被误读”。

## 30 秒摘要

| 这份文档是什么 | 这份文档不是什么 |
|---|---|
| 基于当前仓库的源码白皮书 | 官方产品文档 |
| 面向维护者、逆向研究者、深度用户 | 入门式操作教程 |
| 强调 active path、gated path、drift、failure mode | 功能亮点清单 |

## 为什么要写

这个仓库有三个阅读难点：

1. 它是反编译与重建工程，树上有很多“存在但默认不活跃”的能力
2. 根级说明、docs 站总览页和当前代码之间存在若干漂移
3. 真正值得理解的复杂度并不只在 `query.ts`，而在控制平面、扩展装配和恢复路径

如果没有一套“先分清哪些会跑，再分清为什么这样做”的材料，读者很容易：

- 把 feature-gated 分支当作当前产品说明
- 把 REPL 路径和 headless 路径混为一谈
- 高估某个模块的重要性，低估横切控制平面的难度

## 逆向过程中最值得关注的设计决策

### 1. `feature() => false` 是阅读全局的第一原则

`src/entrypoints/cli.tsx` 直接把 `feature()` polyfill 成恒 `false`。这不是小细节，而是决定“哪些代码只是产品方向线索，哪些才是 external build 活跃路径”的总开关。

### 2. interactive 和 headless 不是一个编排层

很多类似产品会用一个统一 orchestrator 接住所有模式；这里不是。REPL 直连 `query()`，而 print/SDK 路径由 `QueryEngine.ts` 包装。这让很多“总览图”天然容易写错。

### 3. 控制平面比主循环更容易决定维护成本

`AppState`、queue、sessionStorage、task framework、hooks 这些横切层，决定的是“这个系统能不能稳定跑很久”，而不只是“当前一轮会不会返回答案”。

### 4. Markdown workflow 被提升为一等执行能力

skills 不是一堆附属文档，而是通过 frontmatter、fork/inline 执行、权限白名单和动态发现机制被真正接入主系统的能力面。

### 5. 安全体系不是单层拦截，而是纵深拼装

System Prompt 约束、权限规则、沙箱、Plan Mode、Hooks 和预算控制共同构成安全边界。单独看任何一层都会低估它的设计取舍。

## 本站的写作边界

<Warning>
凡是本仓库里无法直接落回源码的结论，都必须明确写成推断，而不是事实。
</Warning>

本站统一遵守这些边界：

- 公开术语和产品名可以参考官方文档
- 逆向结论必须落回当前仓库代码、分析稿或可验证的运行路径
- feature-gated、ant-only、stubbed/removed 必须明确分开
- 不把隐藏能力写成“今天就能用”
